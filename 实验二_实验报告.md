## 四、实验目的与要求
通过本次实验，将体验一种接近企业真实开发的新方式：使用 Codex 协助完成项目开发，在一个完整项目中学习前端、后端、数据库和 AI 集成，理解什么是企业级项目、什么是工程化开发，学会在开发过程中与 AI 协同，而不是只依赖 AI 直接产出代码。

完成本实验后，能够：
1. 理解企业级全栈项目的基本结构。
2. 使用 Spring Boot 开发后端接口。
3. 使用 Vue3 开发前端页面。
4. 使用 MySQL 保存业务数据。
5. 接入大模型，实现聊天、会话、知识库问答等功能。
6. 使用 Git 分支完成协作开发。
7. 能够说明 Codex 在开发中的实际作用。

## 五、实验原理与内容
本实验在实验一搭建的 Spring Boot + MyBatis-Plus 后台基础上，进一步扩展为企业级全栈 AI Agent 项目，核心内容包括：

1. **RAG 知识库增强检索**：在 MySQL 中建立 `kb_document` 知识文档表，用户提问时后端自动检索相关知识文档，将匹配内容作为上下文注入 Prompt，使大模型能够基于私有知识库回答问题，实现"检索增强生成"。
2. **会话上下文管理**：建立 `chat_session`（会话表）和 `chat_message`（消息表），每个对话属于一个会话，多轮对话中后端自动携带最近 N 条历史消息作为上下文，使大模型具备"记忆"能力。
3. **企业级全栈架构**：后端 Spring Boot + MyBatis-Plus + Spring AI + DeepSeek，前端 Vue3 + Vite，前后端分离开发，通过 Vite 代理解决跨域问题。
4. **Git + GitHub 协作开发**：使用 Git 进行版本管理，在开发过程中利用 Codex AI 辅助编码。

## 八、操作异常问题与解决方案

### 问题 1：MyBatis-Plus 实体类字段名与数据库列名不匹配导致查询返回 null

**现象**：`ChatMessage` 实体中的 `sessionId` 字段查询结果始终为 null，但数据库中有数据。

**原因**：MyBatis-Plus 默认将驼峰命名的 Java 字段（`sessionId`）映射为下划线命名的数据库列（`session_id`），但数据库列名和 Java 字段名能够正确对应。经排查发现是 `LambdaQueryWrapper` 中使用了 `ChatMessage::getSessionId` 的方法引用，由于项目未使用 Lombok，getter 方法名正确，但 IDE 自动生成的 getter 可能导致映射问题。

**解决方案**：确认实体类手动编写的 getter/setter 方法名与字段名一致（如 `getSessionId()` 对应字段 `sessionId`），并在 `LambdaQueryWrapper` 中使用 `ChatMessage::getSessionId` 而非手写字符串列名，确保编译期类型安全。最终查询正常返回数据。

### 问题 2：Spring AI 的 ChatModel 对象注入问题

**现象**：在 `AiServiceImpl` 中需要同时注入多个 Service（`ChatSessionService`、`ChatMessageService`、`KbDocumentService`）和 AI 的 `ChatModel`，如果直接注入 `ChatModel` 当未配置 API Key 时会导致启动失败。

**原因**：Spring AI 的 `ChatModel` Bean 在缺少 API Key 配置时可能不存在，直接使用 `@Autowired` 会导致启动报错。

**解决方案**：使用 `ObjectProvider<ChatModel>` 代替直接注入（代码在 `AiServiceImpl.java` 第 `23` 行），通过 `chatModelProvider.getIfAvailable()` 方法（第 `50` 行）获取 Bean，当配置缺失时返回 null 并给出友好提示，不会导致应用启动失败。这保证了即使在最小环境（`application-min.yml` 关闭 AI）中应用也能正常启动。

### 问题 3：前后端分离跨域问题

**现象**：Vue3 前端在 `localhost:3000` 运行，直接请求 `localhost:8080/api/...` 时浏览器报 CORS 跨域错误，请求被拦截。

**原因**：浏览器的同源策略限制，不同端口号的 `localhost` 被视为不同源。

**解决方案**：使用 Vite 开发服务器的 Proxy 代理功能（`vite.config.js` 第 `8-12` 行），将前端所有 `/api` 路径的请求代理转发到后端 `http://localhost:8080`，同时设置 `changeOrigin: true`，后端感知不到跨域，前端的 `fetch` 请求也无需写完整 URL（只需 `/api/...`）。这是前后端分离开发中标准且推荐的解决方案，比在后端添加 CORS 配置更简洁安全。

## 九、实验总结

本次实验在实验一搭建的 Spring Boot + MyBatis-Plus 后台基础上，成功扩展为一个完整的企业级 AI Agent 全栈项目。主要收获如下：

1. **掌握了 RAG（检索增强生成）的核心原理**：通过在 MySQL 中建立知识库文档表，利用 MyBatis-Plus 的 `LambdaQueryWrapper` 进行关键词检索，将检索到的文档作为上下文注入 Prompt，使大模型能够基于私有知识库回答问题，弥补了大模型无法获取专有领域知识的局限。

2. **实现了多轮对话上下文管理**：通过 `chat_session` 和 `chat_message` 两张表持久化存储对话历史，每次提问时自动携带最近 N 条历史消息作为上下文，使大模型具备"记忆"能力，用户体验显著提升。

3. **学会了 Vue3 + Vite 前端工程化开发**：从零搭建 Vue3 项目，使用组件化思想拆分界面（SessionList、ChatMessage、ChatInput、KnowledgePanel），通过 Vite Proxy 解决跨域，实现了前后端分离的企业级开发模式。

4. **体验了 AI 辅助开发的全流程**：在整个实验过程中使用 Codex 协助编码、调试和文档编写，深刻体会了 AI 工具在提升开发效率方面的实际价值，也学会了与 AI 协同而非盲目依赖的开发方式。

5. **理解了企业级全栈项目的架构**：从数据库设计、后端服务分层（Controller → Service → Mapper → Entity）、前端组件化开发、到 API 联调，完整走通了一个企业级项目的全链路。
