# 实验一 Spring Boot 整合 MyBatis-Plus 的搭建实验报告

## 一、实验课程名称
Java高级应用开发

## 二、实验项目名称
搭建 Spring Boot + MyBatis-Plus 后台

## 三、实验类型
设计型

## 四、实验目的与要求
1. 理解 Spring Boot 架构，掌握 Spring Boot 项目的基本组成。
2. 掌握三层架构模式，能够在项目中划分 `Controller`、`Service`、`Mapper` 和实体类。
3. 学习 MyBatis-Plus 的使用，并使用其自带的增删改查功能完成数据库访问。
4. 使用多环境配置文件管理项目参数。
5. 实现 Restful API 风格接口。
6. 接入 AI 的 API 接口，完成基本调用。

## 五、实验原理与内容
本实验基于 Spring Boot 搭建后台项目，采用三层架构进行代码组织：
1. `Controller` 层负责接收请求和返回结果。
2. `Service` 层负责业务逻辑处理。
3. `Mapper` 层负责数据库访问。
4. `Entity/Pojo` 用于封装数据库表字段。

本实验完成的主要内容如下：
1. 使用 Maven 引入 Spring Boot、MyBatis-Plus、MySQL、Lombok、Spring AI 和 DeepSeek 依赖。
2. 配置多环境文件，完成开发环境、生产环境和最小环境的区分。
3. 编写用户模块三层架构代码，并使用 MyBatis-Plus 自带方法完成增删改查。
4. 编写 AI 接口，通过 Spring AI 调用 DeepSeek。
5. 编写前端测试页面，通过页面调用后端 AI 接口。

## 八、操作异常问题与解决方案
1. 问题：原项目使用的 Spring Boot 版本与 Spring AI 兼容性不足。  
   解决方案：将项目 Spring Boot 版本调整为 `3.4.4`，并保持 MyBatis-Plus 使用 Boot 3 对应 starter。

2. 问题：AI 接口最初直接依赖具体 DeepSeek 模型类，IDEA 出现依赖识别异常。  
   解决方案：改为注入 Spring AI 的通用 `ChatModel` 接口，避免因具体实现类识别问题导致代码报红。

3. 问题：DeepSeek `api-key` 写法错误时，程序会把密钥误认为配置占位符。  
   解决方案：直接在 `application-dev.yml` 中写入正确格式的 `api-key`，或者使用正确的环境变量方式注入。

4. 问题：数据库表结构与实体类字段最初不一致。  
   解决方案：将 `User` 实体类改为适配 `shop.user` 表的字段，包括 `phone`、`username`、`password`、`date` 和 `score`。

5. 问题：Apifox 初次调试时默认带有模板前置 URL，导致请求地址拼接错误。  
   解决方案：将请求地址直接改为 `http://localhost:8080/api/...`，或者在环境配置中修改前置 URL。
